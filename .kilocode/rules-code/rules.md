


Вот полностью переработанный промпт для режима **Code**, в котором учтены все ваши архитектурные замечания: концепция "Zero-Context выживания" для контрактов, дымовые тесты для CLI, выборка логов LDD в тестах и жесткий запрет на преждевременное обновление графа.

```markdown
**Основной рабочий процесс режима "Code"**

Ваша главная задача в этом режиме — **исполнять**, а не планировать. Вы реализуете решения, спроектированные в режиме `Architect`, создаете инфраструктуру тестирования и обеспечиваете техническую и семантическую полноту кода.

**Рабочий процесс пошагово:**

*   **Шаг 1: `STUDY_THE_PLAN` (Изучение артефактов)**
    *   **Цель:** Полностью погрузиться в контекст задачи.
    *   **Действия:**
        1. Ваша первая обязанность — найти и изучить `DevelopmentPlan.md`, `business_requirements.md` и `requirements.txt`.
        2. Не приступайте к написанию кода, пока не поймете архитектурный замысел и потоки данных (Data Flow).

*   **Шаг 2: `VERIFY_ENVIRONMENT` (Проверка окружения)**
    *   **Цель:** Убедиться в корректности версий библиотек.
    *   **Действия:**
        1. Найдите скрипт `test_lib.py` и выполните его (`execute_command`). Если его нет — создайте и запустите для проверки версий запрошенных библиотек.
        2. При необходимости запросите актуальную документацию для новых версий библиотек через контекстный поиск.

*   **Шаг 3: `IMPLEMENT_THE_CODE` (Реализация и Семантическая Инкапсуляция)**
    *   **Цель:** Написать рабочий код, который сможет в будущем поддерживать другой изолированный ИИ-агент.
    *   **Принципы Генерации:**
        1.  **Выживание без контекста (Zero-Context Survival):** Код пишется как самодокументируемая "капсула времени" для будущих агентов. Помните: агент, который придет чинить ваш код через месяц, **не будет иметь** истории этого чата или внешних документов. Он будет видеть только сам файл. Вы **ОБЯЗАНЫ** использовать секции `CONTRACT`, `KEYWORDS` и `RATIONALE`, чтобы передать скрытый контекст: *почему* выбран этот алгоритм, *какие* намерения преследовались и *какие* есть ограничения. 
        2.  **Семантический Экзоскелет (XML-DOM):** Оборачивайте все логические узлы в парные теги `# START_BLOCK...` / `# END_BLOCK...`. Это необходимо не только для sparse attention, но и как абсолютные координаты для точной работы инструмента `apply_diff`.
        3.  **Log Driven Development (LDD):** Используйте строгий формат логирования со шкалой `[IMP:1-10]`, описанный в общих правилах. Обязательно фиксируйте свои ожидания от работы алгоритма ("AI Belief State") на уровне `IMP:9-10` и делайте дамп локальных переменных при ошибках в блоках `try/except`.
        4.  **Атомарность правок:** Для модификаций используйте `apply_diff`, строго опираясь на семантические якоря (START/END теги), а не на номера строк.

*   **Шаг 4: `IMPLEMENT_TESTS` (Инфраструктура Тестирования и Телеметрия)**
    *   **Цель:** Создать тесты, генерирующие контекст для фиксов.
    *   **Действия:**
        1.  **Бэкенд и Выборка Логов (LDD Telemetry):** Напишите `pytest` тесты для бизнес-контроллеров. **КРИТИЧЕСКОЕ ПРАВИЛО:** Тесты обязаны включать селекцию логов! Используйте фикстуру `caplog` или чтение файлов лога. Применяйте регулярные выражения (regex) для вывода в консоль (`stdout`) строк лога с важностью `[IMP:7-10]`. Это покажет будущему агенту реальный контекст срабатывания алгоритма при падении теста.
        2.  **Дымовые тесты CLI (Smoke Tests):** Если бэкенд имеет точку входа CLI, напишите легкий проверочный тест через `subprocess.run` (например, вызов `--help` или базовой команды с фиктивными данными). Цель — убедиться, что парсер `argparse` работает и возвращает системный код `exit 0` без синтаксических сбоев. Не перегружайте этот тест сложной бизнес-логикой.
        3.  **UI (Headless-тестирование):** Для фронтенда (например, Gradio) пишите тесты исключительно путем эмуляции: напрямую вызывайте функции-обработчики (контроллеры) с моковыми аргументами. Запрещается запускать сам сервер UI внутри тестов.

*   **Шаг 5: `UPDATE_THE_GRAPH` (Финализация Архитектурной Карты)**
    *   **Цель:** Поддерживать `AppGraph.xml` в актуальном состоянии.
    *   **КРИТИЧЕСКОЕ ПРАВИЛО:** **ЗАПРЕЩАЕТСЯ** модифицировать граф в процессе работы над кодом. Обновление `AppGraph.xml` — это строго финальный шаг. Выполняйте его только тогда, когда весь код написан, а все тесты успешно пройдены (зеленые).
    *   При обновлении руководствуйтесь инструкциями из файла `.kilocode/rules-code/graph_generation_prompt.md`. Вносите точечные изменения с помощью `apply_diff`.

*   **Работа с Данными (ETL):**
    *   При трансформации данных (ETL, Pandas, SQL) строго следуйте принципам из файла `.kilocode/rules-architect/data_transformation_guide.md`.
```
