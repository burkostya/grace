# Инструкция для AI-агента: Создание прототипа ERP-интерфейса на Dash (Lesson_X)

## Общая задача
Спроектируйте и реализуйте прототип ERP-приложения в директории `lesson_X` (вычислите следующий номер урока на основе существующих папок, не читая их содержимое). 
Приложение эмулирует работу с накладными. **Главная архитектурная особенность этой итерации:** полный отказ от выпадающих списков (`agSelectCellEditor`) внутри таблиц в пользу отдельного статического блока поиска товаров. Это гарантирует 100% надежность при работе с номенклатурой от 100 000 позиций.

**UI-ФРЕЙМВОРК:** Dash + dash-bootstrap-components + dash-ag-grid.
**ОКРУЖЕНИЕ:** Без создания новых VENV. Использовать глобальные пакеты. Полная инкапсуляция в `lesson_X/`.

## Архитектура и Разделение слоёв (LDD & Zero-Context)

```
lesson_X/
├── app.py              # UI и Callbacks (только маршрутизация данных)
├── handlers.py         # Чистая бизнес-логика (БЕЗ импорта dash!)
├── db_manager.py       # CRUD операции (SQLite)
├── erp_base.db         # База данных (invoices, items, invoice_lines)
└── tests/
    ├── test_handlers.py    # Тесты бизнес-логики (pytest)
    └── test_callbacks.py   # Тесты коллбеков (передача mock-данных)
```
*Критическое требование:* `handlers.py` не знает о существовании веба. Он принимает базовые типы Python и возвращает базовые типы (словари, списки).

## Структура Базы Данных (`db_manager.py`)
Нужно реализовать 3 таблицы:
1. `invoices` (id, date, client_name, total_sum)
2. `items` (id, sku, name, price) — справочник товаров (создать ~20 демо-позиций при инициализации).
3. `invoice_lines` (id, invoice_id, item_id, qty, price, line_sum)

## Бизнес-требования и Функциональность UI

Интерфейс строится на `dbc.Container(fluid=True)` и состоит из Верхней панели (кнопки "Сгенерировать демо-данные" и "Показать/Скрыть Аналитику") и Рабочей области из **3 колонок**:

### 1. Левая колонка (Список накладных - Master)
*   `dag.AgGrid` со списком накладных (`invoices`).
*   Только чтение, `rowSelection: "single"`.

### 2. Центральная колонка (Search & Detail - 100% Reliability Pattern)
Разделена по вертикали на два логических блока (использовать `dbc.Card` для визуального разделения):

**БЛОК А: Поиск и добавление товара (Работает только если слева выбрана накладная)**
*   Строка: `dbc.Input` (Поиск по наименованию товара) + `dbc.Button` ("Найти").
*   Мини-грид `dag.AgGrid` (Результаты поиска). Колонки: `SKU`, `Наименование`, `Цена`. `rowSelection: "single"`. Высота грида небольшая (например, 200px).
*   Строка под мини-гридом: `dbc.Input` (type="number", placeholder="Кол-во", value=1) + `dbc.Button` ("Добавить в накладную", color="success").

**БЛОК Б: Строки накладной (Detail)**
*   `dag.AgGrid` для отображения `invoice_lines` выбранной накладной.
*   Колонки: `Наименование`, `Количество` (editable: True), `Цена` (readonly), `Сумма` (readonly).
*   *Важно:* Никаких выпадающих списков. Добавление новых строк идет только через Блок А.

### 3. Правая колонка (Аналитика)
*   `dcc.Graph` (Bar Chart: Сумма по каждой накладной). Обновляется при любом изменении сумм.

### Логика взаимодействия (Dash Callbacks)
1. **Поиск:** Ввод текста в поиск -> клик "Найти" -> вызов `handlers.search_items(keyword)` -> обновление мини-грида результатов. (Поиск через SQL `LIKE %keyword%`).
2. **Добавление:** Выделение строки в мини-гриде + ввод количества -> клик "Добавить" -> вызов `handlers.add_line_to_invoice(inv_id, item_id, qty)` (пересчитывает сумму накладной в БД) -> обновление грида накладной (Блок Б), левого грида (обновилась общая сумма) и графика.
3. **Редактирование:** Изменение "Количества" в гриде накладной (Блок Б) вызывает `handlers.update_line_qty(...)` -> пересчет БД -> обновление всех гридов и графика.

1. Код сегментируется тегами `START_BLOCK`, `END_BLOCK`, `MODULE_CONTRACT`.
2. Внедрить LDD 2.0: логирование в `lesson_X/app.log` (Шкала `[IMP:1-10]`, запись состояния автоматов).
3. **Тесты (100% покрытие):**
   * В `test_handlers.py`: протестировать функцию `search_items` (что возвращает правильный список словарей при частичном совпадении). Протестировать `add_line_to_invoice` (что корректно линкует товар к накладной и считает сумму).
   * В `test_callbacks.py`: протестировать логику блокировки кнопок (например, кнопка "Добавить в накладную" должна быть `disabled`, если не выделен товар в мини-гриде поиска).
